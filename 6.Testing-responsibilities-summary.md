# Testing Responsibilities in Modern Software Development

## Overview

This document outlines who writes different types of tests and how they fit into the CI/CD pipeline.

## Test Types and Ownership

### Unit Tests
**Written by:** Developers (as part of feature development)

**Characteristics:**
- Included in pull requests
- Test individual functions/components in isolation
- Fast execution (seconds to minutes)
- Run on every PR

### Integration Tests
**Written by:** Developers and QA Engineers
- Developers: Critical integration points
- QA: Broader integration scenarios

**Characteristics:**
- Test interactions between multiple components/services
- Moderate execution time
- Run after unit tests in CI/CD pipeline

### E2E (End-to-End) Tests
**Written by:** QA Engineers and Test Automation Engineers (with developer contributions)

**Characteristics:**
- Test complete user workflows across the application
- Slow execution (minutes to hours)
- Require fully deployed application
- Run on staging/test environments

|-----------|----------|-----------|-----------------|------------------|

| **E2E** | After merge/deployment | Staging environment | âŒ No (informational) | Minutes to hours |

---

## Why E2E Tests Don't Block PRs

1. **Speed**: Too slow for rapid development cycle
2. **Infrastructure**: Require fully deployed application instance
3. **Cost**: Expensive to run on every PR
4. **Flakiness**: Often fail due to environmental issues rather than code problems
5. **Scope**: Test entire user workflows, not individual changes
